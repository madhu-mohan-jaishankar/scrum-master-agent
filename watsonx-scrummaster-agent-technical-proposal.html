Tier 1 — Repository Sub-Agents

Responsibilities

Each sub-agent is scoped to a single repository or task domain. It polls or receives webhooks for: new commits (author, message, files changed), pull request lifecycle events (opened, review requested, approved, merged, stale), CI/CD pipeline outcomes (pass, fail, flaky), and inline PR comments. It maintains a lightweight local state object containing open PRs, their age, CI status, last commit timestamp, and review coverage. This raw structured state is the sole output pushed upward to assigned Person-Level Master Agents.

WatsonX at Tier 1: Sub-agents invoke watsonx.ai for lightweight classification tasks — distinguishing trivial typo commits from architectural changes, labelling PR review comments as "blocking concern" vs. "nit", and scoring CI flakiness patterns. Small, fast foundation model calls keep latency low.

Tier 2 — Person-Level Master Agents

Responsibilities

One Master Agent exists per team member. It subscribes to the state feeds from all repo sub-agents where that person is active — as author, reviewer, or assignee. Its jobs are: (1) aggregate yesterday's activity into a natural-language standup summary, (2) track real-time WIP count against the team's configured WIP limit, (3) maintain a rolling velocity vector from previous sprints, and (4) detect anomalies such as a person going silent for more than one workday mid-sprint.

WatsonX at Tier 2: The person-level standup digest is generated by a watsonx.ai Granite or Llama 3 model call with a structured prompt: raw activity JSON → concise first-person standup bullet points. Sentiment analysis on PR comment threads surfaces whether a person is blocked due to a genuine blocker or just awaiting review.

Tier 3 — ScrumMaster Agent (Orchestrator)

Responsibilities

The top-level agent receives processed summaries from all Person-Level agents and acts as the true ScrumMaster. It reconciles sprint board state against Jira/Linear/GitHub Projects, calculates burndown against target, runs scope creep detection, nags on stale PRs, triggers ceremony prompts, drafts release notes, and generates a pre-standup briefing sent to Slack or Teams.

WatsonX at Tier 3: The orchestrator uses larger foundation model calls for high-quality prose generation: sprint health narrative, retro sentiment clustering, sprint planning capacity recommendations, and scope creep explanation. watsonx Orchestrate tool-calling handles Jira API writes, Slack message dispatch, and calendar event creation.

3 Core Feature Set

The complete ScrumMaster Agent capability surface — covering sprint health, ceremony automation, and developer experience. Green = will build; Yellow = if time permits.

Sprint Health Intelligence

📉 Burndown Trend Forecasting Must

Linear regression over daily story point completion rate, projected to sprint end. Proactive nudge: "At current pace, 3 tickets won't be done by Friday."

🔴 Stalled vs. Blocked Must

Stalled = no activity + no stated blocker. Blocked = explicit dependency or review waiting. Different escalation paths and message tones for each.

🔔 Meaningful Alerts Must

Threshold-configurable Slack/Teams/email alerts: stale PRs, CI failure streaks, WIP limit breaches, sprint end approaching with points still open.

📋 Standup Digest Per Person Must

LLM-generated per-person standup from commits/PR activity/comments. Progress bar comparing current sprint progress to previous standup snapshot (Mebin's feature).

🔭 Scope Creep Detection Must

Flags tickets added mid-sprint beyond threshold (>10% story points after day 1). Keeps the sprint board honest — synced with what actually happened in GitHub.

⏰ Stale PR Nagging Must

PR approved but not merged for 2+ days → context-aware ping with PR title, author, reviewer(s), age, and CI status. Escalates after second threshold.

🗓 Pre-Standup Brief Must

Sent 15 minutes before standup: all commits, merges, tickets updated, blockers, and the day's sprint health score since yesterday's standup.

🔄 Retro Auto-Drafting Must

Synthesises sprint sentiment from PR comment threads + anonymous survey. Structures into went well / improve / action items with quoted examples.

📦 Story Breakdown Assist Stretch

Flags stories larger than 8 story points and suggests how to break them down. Before creating a new ticket, checks if similar work already exists.

🤝 Cross-Team Blocker Scan Stretch

Scans backlog for tickets that overlap across teams, surfacing external blockers before they become sprint problems.

📅 Holiday / PTO Capacity Stretch

Factors public holidays and PTO into sprint capacity estimates, giving a realistic "will we finish?" forecast adjusted for actual availability.

🆕 New Hire Onboarding Stretch

"What's going on" feature: generates a brief from previous meeting talking points, open PRs, and in-progress tickets. Suggests good-first-issue tickets (Yosief's feature).

Ceremony Facilitation

🗣 Daily Standup (Async) Must

Collects async Slack/Teams updates, surfaces blockers, flags members who haven't updated, and generates a standup summary digest merged with auto-derived activity.

📅 Sprint Planning Assist Must

Pulls velocity from previous N sprints, suggests capacity range, warns about overcommitment, and recommends which backlog tickets fit the sprint.

💬 Retrospective Facilitation Must

Anonymous survey → watsonx.ai sentiment clustering → themes grouped under went well / improve, with representative anonymised quotes and tracked action items.

🚀 Sprint Review Release Notes Must

Auto-generates stakeholder-facing release notes from completed tickets and merged PRs. Auto-generates meeting agenda from current tasks/PRs/issues (Yosief's feature).

5 WatsonX Capabilities Leveraged

IBM WatsonX provides the AI backbone at every layer — from low-latency classification at the repo level to high-quality prose generation at the orchestration level.

watsonx.ai — Foundation Models

Granite / Llama 3 models power all natural language generation: standup digests, sprint health narratives, retro drafts, release notes, and alert messages. Smaller models for high-frequency classification; larger for ceremony output.

watsonx Orchestrate

Used at Tier 3 to define tool-calling workflows: Jira ticket update, Slack message post, Linear sprint state read, calendar event creation. Orchestrate's agent framework maps directly to the ScrumMaster orchestrator role.

watsonx.ai Prompt Lab

Sprint planning and retro prompts are versioned in Prompt Lab, letting non-developer team members tune tone and format without code changes.

RAG Pattern

Sprint Context (open tickets, PR metadata, team velocity, historical data) is injected as retrieval-augmented context into foundation model calls, grounding LLM output in real sprint data.

watsonx.ai Embeddings

PR comment threads and retro survey responses are embedded and clustered to identify recurring themes — used for retro sentiment clustering and team-wide vs. individual blocker detection.

watsonx.data (Optional)

For full-stack WatsonX deployments: historical sprint data and velocity metrics stored in watsonx.data, enabling richer cross-sprint trend analysis via Presto/SQL queries fed into prompts.

Why WatsonX: Enterprise-grade data governance, on-premises deployment options, and a unified platform spanning model inference, orchestration, and data. Granite models are optimised for instruction-following at low latency — ideal for the high-frequency classification tasks at Tier 1.

The WatsonX ScrumMaster Agent is an AI-powered system that acts as a fully autonomous Scrum 
Master for software development teams. It continuously monitors every aspect of a team's sprint 
— from code commits and pull requests on GitHub to task boards in Jira or Linear — and uses 
IBM WatsonX AI to turn that raw activity into clear, actionable intelligence.

The system is built as a three-tier hierarchy of intelligent agents. At the bottom level, 
specialised agents watch each code repository, tracking the health of pull requests, build 
pipelines, and code reviews in real time. One level up, each team member has their own 
personal agent that synthesises all of their activity into a concise daily standup update — 
automatically, without the developer having to write anything. At the top, a single 
ScrumMaster Agent pulls everything together: it monitors sprint progress, spots scope creep 
before it becomes a problem, nudges the team about stale pull requests, and facilitates every 
Scrum ceremony — from sprint planning with data-driven capacity recommendations to 
retrospectives with anonymised sentiment analysis.

IBM WatsonX AI's foundation models generate the natural-language summaries, health reports, 
and retrospective drafts. WatsonX Orchestrate handles the tool-calling workflows that write 
back to Jira, post to Slack, and send email digests. The result is a system that saves 
engineering teams hours of ceremony overhead each sprint, surfaces problems earlier, and 
makes the Scrum process genuinely intelligent — not just automated.
