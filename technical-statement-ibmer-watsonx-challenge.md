# Technical Statement — IBMer watsonx Challenge 2026

## How the Solution Works

The WatsonX ScrumMaster Agent is an AI-powered agent that treats the sprint board as a source of truth and continuously reconciles it against what is actually happening across GitHub, CI pipelines, Jira, and team communication channels. It surfaces that intelligence proactively: a per-person standup digest — derived from commits, PR activity, and ticket updates — arrives 15 minutes before the daily meeting. Stale PR alerts fire when a merged-but-not-closed PR has been sitting for two days. A scope creep warning triggers when new tickets are added mid-sprint beyond the agreed threshold. Sprint planning becomes more accurate because the agent pulls historical velocity data and adjusts available capacity for public holidays within the sprint window. At sprint close, it drafts a structured retrospective from the sprint's actual event history, not from memory. Users interact through the tools they already use — Bob or watsonx for digests and queries, Jira or GitHub Projects for board reconciliation — with no new tooling required.

## How IBM Bob Was Used

IBM Bob was used at every stage of the engineering process, not as a one-off writing tool but as a practical collaborator throughout.

**Idea capture.** The project started with team brainstorm notes: unstructured input mixing pain points and overlapping observations from nine contributors. Bob synthesised these into a structured problem statement and prioritised feature list that became the foundation for all subsequent work.

**Architecture design.** From the confirmed problem statement, Bob generated the architecture proposal covering all five agent modules (Board Reconciliation Engine, Blocker Detection Module, Velocity and Capacity Tracker, Standup Digest Generator, Retrospective Drafter), data flow descriptions, a comparison of architectural alternatives, and the JSON schema for the WatsonX function-calling interface — implementation-ready artefacts produced from plain-language descriptions in a single session.

**Sprint planning.** Using the architecture as input, Bob produced the sprint plan and sub-task breakdown: role cards for each of the nine team members, a critical path analysis, dependency-ordered sub-tasks with explicit acceptance criteria, standup prompt templates, a risk register with named owners, and a holiday-adjusted capacity calculation.

**Prototyping.** Bob scaffolded the first prototype module for the Sprint Capacity Holiday Reduction PoC — Python module stubs with type annotations, unit test skeletons with mocked WatsonX HTTP calls, and the WatsonX function-calling JSON schema — directly from the task breakdown document.

**Documentation and submission.** Bob was given the artifacts from the prior stages and asked to produce the submission package from them: the revised architecture document, the submission planning meeting agenda, the Solution Statement, and this Technical Statement — generated by Bob reading its own prior outputs and describing its role at each stage.

That last step is the most distinctive use of Bob in this project. Bob's ability to hold full project context across a session and produce structured documentation from its own prior outputs is the same capability the ScrumMaster Agent applies when it drafts a retrospective from a sprint's actual event history rather than from memory.
